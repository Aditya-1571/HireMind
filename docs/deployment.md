# HireMind Deployment Guide

This guide prepares HireMind for a split deployment:

- Frontend on Vercel
- Backend on Render, Railway, Fly.io, or another Python web platform
- PostgreSQL on Neon
- Local Ollama on the machine that runs the backend, unless you intentionally
  expose a private Ollama host

Do not commit real secrets.

## Frontend Deployment: Vercel

Root directory:

```text
frontend
```

Build command:

```bash
npm run build
```

Install command:

```bash
npm ci
```

Runtime command:

```bash
npm run start
```

Required Vercel environment variables:

```env
FRONTEND_APP_ENV=production
BACKEND_API_URL=https://your-backend.example.com
NEXT_PUBLIC_GOOGLE_CLIENT_ID=your-google-client-id.apps.googleusercontent.com
SITE_URL=https://your-frontend.example.com
SESSION_COOKIE_NAME=hiremind_session
```

Notes:

- `BACKEND_API_URL` is server-only and must not use `NEXT_PUBLIC_`.
- `NEXT_PUBLIC_GOOGLE_CLIENT_ID` is safe to expose because Google Identity
  Services requires it in the browser.
- `SITE_URL` controls canonical metadata, robots, sitemap, and social-sharing
  URLs.
- `SESSION_COOKIE_NAME` is optional; keep the default unless you have a naming
  conflict.
- Production builds should not depend on localhost.
- Metadata, robots, sitemap, manifest, Open Graph, and Twitter card assets are
  generated by the Next.js app.

## Backend Deployment: Render, Railway, or Fly.io

Root directory:

```text
backend
```

Install command:

```bash
pip install -r requirements.txt
```

Recommended release/migration command:

```bash
alembic upgrade head
```

Startup command:

```bash
uvicorn app.main:app --host 0.0.0.0 --port $PORT
```

Required backend environment variables:

```env
BACKEND_APP_ENV=production
BACKEND_APP_NAME=HireMind API
BACKEND_CORS_ORIGINS=["https://your-frontend.example.com"]
BACKEND_TRUSTED_HOSTS=["your-backend.example.com"]
DATABASE_URL=postgresql+psycopg://username:password@host/database?sslmode=require
GOOGLE_CLIENT_ID=your-google-client-id.apps.googleusercontent.com
SESSION_SECRET=replace-with-a-long-random-production-secret
SESSION_EXPIRE_MINUTES=1440
OLLAMA_BASE_URL=http://127.0.0.1:11434
OLLAMA_MODEL=qwen2.5:1.5b
OLLAMA_TIMEOUT_SECONDS=60
```

The backend settings support normal names and selected `BACKEND_`-prefixed
aliases, for example `DATABASE_URL` or `BACKEND_DATABASE_URL`.

Production safeguards:

- `BACKEND_APP_ENV=production` requires a strong `SESSION_SECRET`, configured
  Google client ID, configured database URL, explicit CORS origins, and explicit
  trusted hosts.
- Do not use wildcard CORS with credentials.
- Keep the backend behind HTTPS.
- Configure platform-level request and upload size limits where available.

## Neon PostgreSQL

Use a Neon PostgreSQL connection string with SSL:

```env
DATABASE_URL=postgresql+psycopg://username:password@host/database?sslmode=require
```

Before launch:

```bash
cd backend
alembic upgrade head
alembic current
```

The application uses SQLAlchemy `pool_pre_ping=True` for stale connection
handling.

## Google OAuth

Configure the production OAuth client in Google Cloud Console.

Authorized JavaScript origins:

```text
https://your-frontend.example.com
http://localhost:3000
```

Redirect URIs:

HireMind uses Google Identity Services token flow through the frontend and
backend API, not a traditional redirect callback. If Google requires redirect
URIs for your OAuth client type, include your frontend origin and login page.

Use the same `GOOGLE_CLIENT_ID` on the backend and
`NEXT_PUBLIC_GOOGLE_CLIENT_ID` on the frontend.

Never place `DATABASE_URL`, `SESSION_SECRET`, or private backend URLs in
`NEXT_PUBLIC_` variables.

## Health Checks

Backend:

- `GET /api/health`
- `GET /api/health/database`

Use `/api/health` as the platform health check. It is lightweight and does not
require database access.

Use `/api/health/database` for operational diagnostics. It returns only safe
status information.

## CI/CD

GitHub Actions are included for validation only:

- `.github/workflows/frontend-ci.yml`
- `.github/workflows/backend-ci.yml`

They do not deploy automatically.

## Common Deployment Issues

- Frontend build fails with `BACKEND_API_URL is required in production`:
  set `FRONTEND_APP_ENV=production` only when production variables are present.
- Google sign-in fails:
  confirm OAuth JavaScript origins and matching client IDs.
- Browser blocks API calls:
  confirm `BACKEND_CORS_ORIGINS` includes the exact frontend origin.
- Backend rejects host:
  confirm `BACKEND_TRUSTED_HOSTS` includes the backend host sent by the platform.
- Database health fails:
  confirm Neon connection string, SSL mode, migrations, and IP/network access.
- Ollama unavailable:
  confirm the backend can reach `OLLAMA_BASE_URL` and the configured model is
  installed.

## Production Launch Checklist

- Frontend production build passes.
- Backend compile, tests, and import check pass.
- `alembic upgrade head` has run against production database.
- `SESSION_SECRET` is strong and private.
- `BACKEND_CORS_ORIGINS` contains only production frontend origins.
- `BACKEND_TRUSTED_HOSTS` contains only production backend hosts.
- Google OAuth production origin is configured.
- HTTPS is enabled for frontend and backend.
- Deployment-layer rate limits are configured.
- Health check endpoint is configured.
- Logs do not include secrets, session tokens, resumes, answers, or prompts.
